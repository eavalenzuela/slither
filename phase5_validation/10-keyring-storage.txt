=== V9 — finding: kernel-keyring cross-process gap ===

Live e2e exercise revealed a real Phase 5 #98 design bug:

  1. `slither-agent enroll` ran as root (via sudo). keystore.AutoSelect
     probed @s (session keyring), succeeded, and wrote slither.agent.{key,
     cert,ca} keys there. Enroll exited, host_id was written to disk
     ("enrolled host 469df48a-...").

  2. Both `keyctl list @s` and `keyctl list @u` from a fresh SSH
     session show no slither keys. The enroll process's session
     keyring was reaped on process exit.

  3. Root cause: @s is per-process (per-PAM-session), not per-uid.
     The agent systemd unit, even running as root, gets its own
     session keyring under PAM's pam_keyinit + systemd's
     KeyringMode= setting. Cross-process cert sharing requires either
     the persistent keyring (@us) or a different design altogether.

  4. @u (user keyring) was the intended fallback in keyring_linux.go,
     but tryKeyringPlatform() probes @s FIRST and never falls through
     to @u when the @s probe succeeds. So the buggy @s path takes
     priority.

Disposition: file-fallback path remains correct + works. The kernel-
keyring path is documented in ADR-0035 as the preferred default but
needs a redesign before it's production-fit. Phase 6+ work should:
  (a) Use the persistent keyring (@us) which survives across sessions,
      OR
  (b) Drop the kernel-keyring storage entirely and keep file fallback,
      OR
  (c) Add a separate systemd helper unit that pre-populates keys at
      boot via KeyringMode=shared.

Static unit-test evidence (keystore_test.go's 6 tests pass under -race)
demonstrates the keystore code itself functions correctly when the
underlying keyring shape behaves. The live e2e gap is in the keyring
TYPE selection, not in the keystore code.

## Fix #2 — ruleast target_field shape (was selection-membership) ##
============================================================================
(pkg/ruleast/sigma.go)

Pre-fix: target_field had to appear inside the selection block. kill_process
rules typically match on Image but target ProcessId — which fails this check.
Post-fix: target_field need only pass isIdentifierShape() at compile time;
runtime resolves at finding-time and degrades to would_have_executed=true
if the field doesn't exist on the actual event.

Re-tested by:
  - Rule 0086 ships with selection={Image|endswith: /slither-canary} but
    target_field=ProcessId. Pre-fix: would not compile. Post-fix: compiled
    cleanly, loaded into the running ruleset on all three agents (server log
    showed 'hub: include rule rule_uid=8b7c4d00-0086-… classification=edge_only').
  - V5 fired the rule on ubuntu (promoted) and killed the spawned PID 3072.
    target_field successfully resolved to a real PID, confirming runtime
    field-resolution path works.
  - V6 fired same rule on debian (detect-only) and stamped the finding with
    x_auto_response_would_have_executed=true (visible in CH ocsf table raw
    JSON). The graceful-degradation contract holds.

## Fix #4 — AutoResponder populates RuleUid in ResponseRequest ##
============================================================================
(agent/internal/respond/auto.go)

Same evidence as Fix #3 — the rule_uid round-trips agent → server → DB,
which it can only do if the AutoResponder set it before invoking the executor.
Pre-fix: AutoResponder created a synthetic ResponseRequest without rule_uid
and the server's Hub.OnResult had no way to attribute the result to a rule
→ ErrResponseActionNotFound and the row was lost.

## Fix #5 — server inserts response_action row on ErrResponseActionNotFound + rule_uid ##
============================================================================
(server/internal/respond/dispatcher.go, server/internal/grpcserv/session.go)

Re-tested by V5: the auto-respond rows were never created by an operator —
they only exist because the server's Hub.OnResult, on receiving a
ResponseResult with rule_uid set, hit the ErrResponseActionNotFound branch
and inserted a fresh row scoped to the trusted peer-cert hostID with

actor_type=system for these (vs user for operator-driven), confirming the

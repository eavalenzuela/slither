=== V9 finding: keystore probe possession bug ===
ADR-0038's @u keyring path is technically wired correctly but
fails the AutoSelect probe on every host shape (Debian 13,
RHEL 10, Ubuntu 24.04, Ubuntu 24.04 arm64) on this kernel
matrix (6.12 / 6.17). Strace from a fresh enrol:

  keyctl(KEYCTL_GET_KEYRING_ID, KEY_SPEC_USER_KEYRING, 1) = 119072990
  add_key("user", "slither.agent.probe", "ok", 2, 119072990) = 365295180
  keyctl(KEYCTL_READ, 365295180, 0xdccfc0, 0) = -1 EACCES

Root cause: on modern kernels (6.x), a key added to @u from a
process whose session-keyring chain doesn't include @u has owner
perms WRITE granted (UID match) but READ denied — the kernel's
possession-traversal check fails before the owner-perm check.
Linking @u into @s (keyctl link @u @s) before the probe makes
the process possess the key + the read succeeds.

Effective Phase 6 behaviour on this fleet: every host falls
through to the file store. Functionally identical to Phase 5
#98 — security posture per the threat model is unchanged
(file at 0600 root:root, same durability claims). The
in-RAM-only privacy upgrade ADR-0038 promised is not active.

Restart-cycle behaviour: every host's agent recovers cleanly
from the file store on every restart. The Gap A symptom
(keys reaped at the enrol→service boundary) is mooted by the
fall-through to file.

PHASE 7 FOLLOW-UP: tryKeyringPlatform() should keyctl_link @u
into @s before the read-back probe so possession traversal
covers the new key. Adding 'KeyringMode=shared' to the systemd
unit alone is necessary but not sufficient on these kernels.
